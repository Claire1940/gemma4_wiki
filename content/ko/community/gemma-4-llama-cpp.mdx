## Gemma 4 모델 제품군 이해하기

Google DeepMind는 Gemma 4를 "모바일 우선" AI 릴리스로 구성했지만, 데스크톱 사용자에게는 더 큰 변형 모델에서 그 아키텍처가 진정으로 빛을 발합니다. 이전 세대와 달리 Gemma 4는 중급 모델에 MoE(Mixture of Experts) 방식을 활용하여 훨씬 더 큰 밀집 모델에 필적하는 높은 "지능 밀도"를 제공합니다.

이 제품군은 "유효" 파라미터 수에 따라 여러 계층으로 나뉘며, 이는 사용자가 자신의 VRAM 제약에 맞는 버전을 결정하는 데 도움이 됩니다.

| 모델 변형 | 유효 파라미터 | 실제 파라미터 | 컨텍스트 윈도우 | 주요 용도 |
| :--- | :--- | :--- | :--- | :--- |
| **Gemma 4 2B** | 20억 개 | 50억 개 | 128k | 모바일 기기 / 기본 채팅 |
| **Gemma 4 4B** | 40억 개 | 80억 개 | 128k | 코딩 / 로컬 RAG |
| **Gemma 4 26B (MoE)** | 260억 개 | 260억 개 | 256k | 복잡한 추론 / 에이전트 |
| **Gemma 4 31B** | 310억 개 | 310억 개 | 256k | 프런티어 연구 / 멀티모달 |

> **💡 팁:** VRAM이 제한적인 경우(12GB 미만), 속도와 지능의 최적의 균형을 위해 4비트 양자화가 적용된 4B 유효 버전을 사용하는 것이 좋습니다.

## Llama.cpp로 Gemma 4 설정하기

**gemma 4 llama cpp**를 실행하려면 최신 빌드의 Llama.cpp 리포지토리가 필요합니다. Google이 2026년에 특정 GGUF 지원이 필요한 새로운 아키텍처 변경 사항을 도입했기 때문입니다. 로컬 서버를 실행하려면 다음 단계를 따르세요.

### 1. Llama.cpp 설치
macOS 사용자의 경우 Homebrew가 여전히 가장 빠른 방법입니다. Linux 및 Windows 사용자는 AVX-512 또는 CUDA 지원이 완전히 최적화되도록 소스에서 컴파일해야 합니다.

```bash
# macOS의 경우
brew install llama.cpp

# Linux/Windows의 경우 (소스에서 빌드)
git clone https://github.com/ggerganov/llama.cpp
cd llama.cpp
make
```

### 2. 양자화된 GGUF 가중치 다운로드
Hugging Face에서 모델 가중치를 가져와야 합니다. 커뮤니티에서는 일반적으로 출시 직후 GGUF 버전을 제공합니다. 16GB RAM 머신의 경우 `IQ2_M` 또는 `Q4_K_M` 양자화가 권장됩니다.

| 양자화 | 필요한 메모리 (26B) | 품질 손실 | 속도 |
| :--- | :--- | :--- | :--- |
| **Q8_0** | ~28 GB | 무시할 수 있음 | 빠름 |
| **Q4_K_M** | ~16 GB | 낮음 | 매우 빠름 |
| **IQ2_M** | ~9 GB | 보통 | 극한 |

### 3. 서버 실행
`.gguf` 파일이 준비되면 Llama.cpp 서버를 시작합니다. 이를 통해 웹 UI 또는 OpenAI 호환 API를 통해 모델과 상호작용할 수 있습니다.

```bash
llama-server --model ./models/gemma-4-26b-it.Q4_K_M.gguf --ctx-size 8192 --port 8080
```

## 하드웨어 성능 및 토큰 속도

**gemma 4 llama cpp** 설정에서의 성능은 메모리 대역폭에 크게 좌우됩니다. Gemma 4는 기본 추론(thinking) 단계를 지원하므로 초기 "생각" 단계는 더 오래 걸릴 수 있지만, 최종 생성은 2026년형 실리콘에서 놀라울 정도로 빠릅니다.

48GB 통합 메모리가 탑재된 Apple M4 Pro에서 26B MoE 모델은 8비트 양자화를 사용하여 초당 약 40-42개의 토큰을 생성합니다. 이는 실시간 읽기 및 코딩 지원에 충분한 속도입니다.

| 하드웨어 구성 | 모델 크기 | 양자화 | 초당 토큰 수 |
| :--- | :--- | :--- | :--- |
| **Apple M4 (16GB)** | 4B Effective | Q8_0 | 65+ t/s |
| **Apple M4 Pro (48GB)** | 26B MoE | Q8_0 | 42 t/s |
| **NVIDIA RTX 4090 (24GB)** | 26B MoE | Q4_K_M | 55 t/s |
| **NVIDIA RTX 5090 (32GB)** | 31B Dense | Q4_K_M | 70+ t/s |

> **⚠️ 경고:** VRAM이 8GB에 불과한 GPU에서 26B 또는 31B 모델을 실행하려고 하면 시스템이 가중치를 시스템 RAM으로 오프로드(System Fallback)하므로 속도가 극도로 느려집니다.

## 2026년의 멀티모달 기능

Gemma 4의 눈에 띄는 특징 중 하나는 기본 멀티모달 학습입니다. "비전 인코더"를 나중에 추가한 이전 모델들과 달리, Gemma 4는 이미지, 오디오, 심지어 비디오까지 이해할 수 있도록 처음부터 학습되었습니다.

로컬 테스트에서 **gemma 4 llama cpp** 조합은 다음 작업에 탁월합니다:
1.  **객체 탐지:** 사진에서 특정 UI 요소나 항목을 식별하고 바운딩 박스를 제공합니다.
2.  **복잡한 OCR:** 지저분한 영수증이나 손글씨 메모를 거의 완벽한 정확도로 읽습니다.
3.  **UI 기반 코드 생성:** 웹사이트 스크린샷을 찍어 기능적인 HTML/CSS/React 코드로 재현합니다.
4.  **다이어그램 분석:** 소프트웨어 아키텍처 다이어그램(예: Clean Architecture 또는 Hexagonal)을 해석하고 데이터 흐름을 설명합니다.

## 고급 에이전트 워크플로우

Gemma 4는 에이전트 동작에 최적화되어 있으며, 특히 시스템 지침을 따르고 외부 도구를 호출하도록 튜닝되었습니다. Llama.cpp를 통해 실행할 때 구조화된 JSON 출력을 활용하여 모델이 로컬 스크립트와 완벽하게 통신하도록 할 수 있습니다.

Google은 이번 버전에서 "기본 시스템 지침" 준수 능력을 향상시켰습니다. 즉, 모델에게 "터미널처럼 행동하라"고 지시하면 Gemma 2 또는 3에 비해 캐릭터를 유지할 확률이 더 높습니다. 따라서 컨테이너화된 환경에서 CLI 명령을 안전하게 생성하고 실행할 수 있는 로컬 DevOps 에이전트용으로 이상적입니다.

공식 연구 및 라이선스에 대한 자세한 내용은 [Hugging Face](https://huggingface.co/google)를 방문하여 모델 카드 및 Apache 2.0 라이선스 세부 정보를 확인할 수 있습니다.

## FAQ

**Q: 8GB RAM이 장착된 일반 노트북에서 Gemma 4를 실행할 수 있나요?**

A: 네, 하지만 "유효 2B" 파라미터 버전으로 제한됩니다. 사용 가능한 추론 속도를 유지하면서 모델이 메모리 제한 내에 머물도록 `IQ2_M` 또는 `Q3_K_S`와 같은 고압축 양자화를 사용해야 합니다.

**Q: gemma 4 llama cpp는 GPU 가속을 지원하나요?**

A: 물론입니다. Llama.cpp는 CUDA (NVIDIA), Metal (Apple), Vulkan (AMD/Intel)을 지원합니다. 서버를 시작할 때 `-ngl`(GPU 레이어 수) 플래그를 사용하여 최대한 많은 레이어를 그래픽 카드로 오프로드하여 성능을 극대화하세요.

**Q: 코딩 면에서 Gemma 4가 Qwen 3.5보다 나은가요?**

A: Qwen 3.5는 순수 로직 면에서 높은 평가를 받지만, Gemma 4는 "사고(thinking)" 작업과 멀티모달 코드 생성(이미지에서 코드 구축)에서 우수한 성능을 보여줍니다. 표준 Python 또는 TypeScript 생성의 경우 둘 다 최상위 수준이지만, Gemma 4의 Apache 2 라이선스가 상업적 용도로 더 매력적인 경우가 많습니다.

**Q: 긴 "추론(reasoning)" 지연 시간은 어떻게 해결하나요?**

A: Gemma 4는 o1 시리즈와 유사한 "사고하는" 모델입니다. 지연 시간이 너무 길면 시스템 프롬프트를 조정하여 "내부 독백 없이 간결한 답변"을 요청하세요. 이렇게 하면 **gemma 4 llama cpp** 설정에서 첫 번째 토큰 생성 시간을 크게 줄일 수 있습니다.
